ENTRY — 2026-06-13 (Unified waypoint teleport delay — v1.3.34)
================================================================================

Version: 1.3.34
Jar: /home/pi/Plugins/Pinpoint/build/libs/Pinpoint-1.3.34.jar

Files Changed:
  MOD  build.gradle.kts
  MOD  src/main/resources/plugin.yml
  MOD  src/main/resources/config.yml
  MOD  changelog.txt
  MOD  chatgpt.txt

--- DESIGN ---
All Waystone/Landmark/Pinpoint destination teleports now always use a 5-second countdown,
regardless of whether triggered via compass/GUI or by right-clicking a placed block.

Previously:
  compass/GUI path → compass-delay-seconds (default 10s)
  block right-click path → block-delay-seconds (5s)

Now:
  Both compass-delay-seconds and block-delay-seconds default to 5.

Player-to-player teleport is completely separate (stand-still-seconds, default 10s) and
was not touched.

--- FEE SAFETY ---
Bops are charged AFTER player.teleport() succeeds in TeleportHelper.doTeleport().
If the countdown is cancelled (player moves, takes damage, quits), the Bukkit task is
cancelled and doTeleport() is never called — so no Bops are ever charged on a failed teleport.
This was already correct and remains unchanged.

--- WHAT WAS NOT CHANGED ---
- TeleportHelper.java — no logic changes, only config default values changed
- GUI/invite behavior — untouched
- Player-to-player teleport flow — untouched
- Message strings — teleport-countdown uses %d so it already shows "5s" automatically

--- TESTING CHECKLIST ---
1. Teleport to a Landmark/Waystone via compass GUI — confirm 5-second countdown in action bar.
2. Right-click a placed Pinpoint block — confirm 5-second countdown in action bar.
3. Teleport to a fee-bearing waypoint (set cost > 0) — confirm Bops deducted only after arriving,
   not at countdown start.
4. Move or take damage during a 5s countdown — confirm "Teleport cancelled" message and no Bops taken.
5. Send a player-to-player /tprequest — confirm it still uses its own stand-still-seconds (10s)
   and its own action bar messages.
6. All messages referencing cost say "Bops", not "money" or "bits".

ENTRY — 2026-06-12 (Context-aware teleport delay — v1.3.33)
================================================================================

Version: 1.3.33
Jar: /home/pi/Plugins/Pinpoint/build/libs/Pinpoint-1.3.33.jar

Files Changed:
  MOD  build.gradle.kts
  MOD  src/main/resources/plugin.yml
  MOD  src/main/resources/config.yml
  MOD  src/main/java/com/pinpoint/util/TeleportHelper.java
  MOD  src/main/java/com/pinpoint/gui/GuiManager.java
  MOD  changelog.txt
  MOD  chatgpt.txt

--- DESIGN ---
Teleport delay is now determined by HOW the player initiated travel:

  Physical block right-click (placed Pinpoint or Landmark block):
    → teleport.block-delay-seconds  (default 5)

  Pinpoint Compass right-click or any GUI tab navigation:
    → teleport.compass-delay-seconds  (default 10)

Both delays apply equally to normal Pinpoints AND Landmarks.

--- ROOT CAUSE OF THE PREVIOUS SILENT BUG ---
GuiManager.openHubGui(player, focusedWaypointId, fromBlock) accepted a fromBlock=true
argument from WaypointInteractListener when a player right-clicked a placed block, but
the method body completely ignored that parameter and always opened the GUI in compass mode.
As a result, teleportFromBlock() (5s path) was dead code through the GUI flow.

--- HOW IT WAS FIXED ---
1. GuiManager gained a per-player Map<UUID, Boolean> playerFromBlock field.
2. openHubGui() now stores playerFromBlock.put(uuid, fromBlock) at entry.
3. onInventoryClose() clears playerFromBlock.remove(uuid) on full GUI close (not on
   sub-GUI navigation, so the flag survives menu-to-menu navigation).
4. handleWaypointClick() looks up the stored flag and passes it to:
     openOwnerViewGui(player, wp, fromBlock)  — new param
     openUseGui(player, wp, fromBlock)        — already had param, now correct
     openLandmarkUseGui(player, wp, fromBlock)  — new param
     openLandmarkManageGui(player, wp, fromBlock)  — new param
5. All "Teleport Here" handlers in those GUIs now dispatch to teleportFromBlock()
   (5s) when fromBlock=true, and teleport() (10s) otherwise.
6. openConfirmLandmarkDeleteGui() gained a fromBlock param so "Cancel" can return
   to openLandmarkManageGui() with the correct context.

--- CONFIG CHANGES ---
Old keys removed:
  settings.waypoint-pearl-teleport-delay-seconds
  settings.waypoint-block-teleport-delay-seconds
  landmarks.teleport.delay-seconds

New unified section added at the top of config.yml:
  teleport:
    compass-delay-seconds: 10
    block-delay-seconds: 5

landmarks.teleport now has a comment pointing to the unified section.

--- UNCHANGED ---
- Player-to-player teleport uses player-teleport.stand-still-seconds (default 10).
- Movement cancel, Bops charging, invincibility window, cooldown all unchanged.
- teleportGroupMember() and partyFollow() are unchanged (always 5s group countdown).

ENTRY — 2026-06-11 (Player teleport fixes — v1.3.32)

--- BUGS FIXED ---
1. Sender received two chat messages when clicking a player in the Players tab
   (player-tp-request-sent + player-tp-request-expires-in). Both removed. The sender
   is now completely silent on request send. Only error messages (player offline,
   already a request pending) are shown to the sender.

2. renderPlayersContent was calling plugin.getConfig().getDouble("settings.player-teleport-cost")
   — a key deleted in v1.3.31. It returned the default 100.0 but then formatted it via
   EconomyManager.format() which may produce garbled currency text. Fixed to read the
   correct key player-teleport.cost-bops and display "N Bops each" literally.

3. player-teleport.cost-bits key renamed to player-teleport.cost-bops. All code updated.

--- DESIGN CHANGES ---
Currency: All player-to-player teleport text now says "Bops" explicitly. EconomyManager
.format() is NOT used for player-to-player teleport display — cost is shown as integer
Bops. Normal waypoint fees still use EconomyManager.format() (Vault currency unchanged).

Balance check timing: Moved from post-teleport to pre-countdown (at accept time).
Both players must have cost-bops Bops before the 10s countdown starts. If either is
short, the accept is rejected with clear per-player messages and no countdown starts.

Post-teleport charge: Still happens after teleport completes. Edge-case failure handling
added: if a player spent Bops during the countdown and can't be charged, the server logs
a WARNING and neither player is charged (requester refunded if target's charge fails).

--- CONFIG CHANGES ---
player-teleport:
  cost-bits → cost-bops  (renamed; same value 100)

New messages (replacing old cost/funds messages):
  player-tp-bops-notice     — shown at accept time: "Cost: %d Bops each on success."
  player-tp-no-bops-self    — shown to the broke player: "You don't have enough Bops (%d needed)."
  player-tp-no-bops-other   — shown to the other player: "%s doesn't have enough Bops (%d needed)."
  player-tp-bops-charged    — shown after successful charge: "%d Bops deducted from each of you."

Removed messages (no longer used):
  player-tp-request-sent      (sender gets nothing on success)
  player-tp-cost-notice       (replaced by player-tp-bops-notice)
  player-tp-insufficient-funds (check moved to pre-countdown; new messages cover it)
  player-tp-charged           (replaced by player-tp-bops-charged)

ENTRY — 2026-06-10 (Landmark = ESO Wayshrine travel terminal — v1.3.29)
================================================================================

--- DESIGN GOAL ---
The Pinpoint system is inspired by Elder Scrolls Online Wayshrines.
A Landmark is a travel terminal. The player is already standing at it.
Offering "Teleport Here" when they right-click makes no sense — they are
physically touching the destination already.
Landmarks should act as access points into the full Pinpoint travel network.

--- PREVIOUS BEHAVIOR (v1.3.28) ---
Right-clicking a Landmark Block opened the full hub GUI, but forced the
Landmarks category tab to be active. When the player then clicked any Landmark
entry in that list (including the one they were standing at), they were taken to
openLandmarkUseGui which presented "Teleport Here" — a screen specific to a
single Landmark. This violated the travel terminal design.

--- NEW BEHAVIOR (v1.3.29) ---
Right-clicking a Landmark Block now opens the full Pinpoint travel network at
the player's last-visited category (defaulting to My Pinpoints / OWNED on first
open). The player immediately sees the full tab bar:
  My Pinpoints | Invited | Public | Players | Landmarks
No Landmark-specific screen appears first. The Landmark is an access point, not
a destination.

--- CODE CHANGE ---
File: GuiManager.java — openHubGui()
The LANDMARK branch inside the fromBlock+focusedWaypointId guard was changed
from:
    playerCategory.put(player.getUniqueId(), Category.LANDMARKS);
    renderCategoryGui(player, Category.LANDMARKS, getCategoryPage(...));
To:
    Category cat = playerCategory.getOrDefault(player.getUniqueId(), Category.OWNED);
    renderCategoryGui(player, cat, getCategoryPage(player.getUniqueId(), cat));

This is a single-path change with zero impact on normal Pinpoint blocks or any
data layer.

--- ADMIN BEHAVIOR ---
Admin management is fully preserved. Admins (pinpoint.admin.landmark) can access
the Landmark Manage GUI by navigating to the Landmarks tab in the hub and
clicking any Landmark entry — handleWaypointClick() routes admins to
openLandmarkManageGui() as before.

--- COMPATIBILITY NOTES ---
- waypoints.yml: no changes, all LANDMARK entries read/write identically.
- Hologram system: unchanged.
- PDC keys: unchanged (pinpoint:landmark_block, pinpoint:waypoint_block).
- Compass distribution: unchanged.
- Teleport implementation: unchanged.
- Bops costs: unchanged.
- Waypoint limits: unchanged.
- Normal Pinpoint Block interaction: unchanged.

--- FEATURE: Dedicated Landmark Block Item ---

ITEM PDC TAG / KEY
  Normal Pinpoint Block: NamespacedKey(plugin, "waypoint_block")  → pinpoint:waypoint_block
  Landmark Block:        NamespacedKey(plugin, "landmark_block")  → pinpoint:landmark_block
  These are completely independent. isWaypointBlockItem() checks only waypoint_block.

ITEM CONSTRUCTION (createLandmarkBlock)
  Material: read from landmarks.block.material (default LODESTONE).
  Display name: read from landmarks.block.name, parsed with MiniMessage.miniMessage().deserialize().
    Italic=false applied. Default: "<gradient:#00e5ff:#8a2be2><bold>Landmark Block</bold></gradient>"
  Lore: read from landmarks.block.lore (StringList), parsed with
    LegacyComponentSerializer.legacyAmpersand().deserialize() per line. Italic=false applied.
    Default: ["&7Admin-only travel landmark.", "&7Place this during landmark setup."]
  PDC: KEY_LANDMARK_BLOCK set to BYTE 1.

COMMAND FLOW (/wp landmark create <name>)
  1. Admin runs /wp landmark create SpawnHub.
  2. Plugin validates: permission, landmarks enabled, name length ≤ 32, no duplicate name.
  3. setPendingLandmarkCreation(uuid, name) registers setup mode.
  4. Checks admin inventory for an existing Landmark Block (isLandmarkBlock scan).
     If none found: addItem(createLandmarkBlock()), sends "A Landmark Block has been added."
  5. Sends "Place the Landmark Block to set the location for landmark 'SpawnHub'."

PLACEMENT BEHAVIOR (BlockPlaceListener — landmark path)
  Triggered when isLandmarkBlock(item) is true.
  Check 1 — permission: requires pinpoint.admin.landmark.create.
    Fail: event.setCancelled(true) + landmarks.block.no-permission-message.
  Check 2 — setup mode: hasPendingLandmarkCreation(uuid) must be true.
    Fail: event.setCancelled(true) + landmarks.block.setup-required-message.
  Check 3 — landmarks enabled: landmarks.enabled must be true.
    Fail: event.setCancelled(true) + "Landmarks are disabled in config."
  Success: clearPendingLandmarkCreation → createLandmark(name, blockLocation) →
           spawnHologram(landmark) → "Landmark 'X' created!" message.
           Block placement is NOT cancelled — the Landmark Block physically occupies the world.

PLACEMENT BEHAVIOR (BlockPlaceListener — normal Pinpoint path)
  Unchanged. Triggered when isWaypointBlockItem(item) is true. The two paths are
  completely separate if/else branches. No interaction between them.

MAX WAYPOINT LIMIT BEHAVIOR
  Landmark creation goes through the landmark branch of BlockPlaceListener, which
  never calls isAtWaypointLimit(). Landmarks use LANDMARK_OWNER_UUID (UUID 0,0)

--- FEATURE: Automatic Pinpoint Compass Distribution ---

Replaced the simple hasPlayedBefore() first-join check with a permanent,
UUID-based "received compass" tracking system that mirrors the GuideBook
plugin's PlayerDataManager pattern.

STORAGE FORMAT
  File: plugins/Pinpoint/compass-received.yml
  Structure:
    players:
      <uuid>: true
      <uuid>: true
  A UUID is present in the file only when the player has received a compass.
  Absence from the file means "not yet received" — no false values written.
  Written on every markReceived() call (immediate, not batched).
  Safe for thousands of players (only truthy entries stored).

JOIN BEHAVIOR
  On every PlayerJoinEvent (MONITOR priority):
  1. Check compass.enabled — if false, do nothing.
  2. Check CompassDataManager.hasReceived(uuid) — if true, do nothing.
  3. Schedule a 1-tick delayed task (inventory fully initialized):
     a. Check if player already has a valid Pinpoint Compass in inventory.
        If yes → markReceived() and return (no duplicate given).
     b. Give compass via addItem(). If inventory has space → normal give,
        show compass.message. If full → dropItemNaturally(), show
        compass.inventory-full message.
     c. markReceived() — write UUID to compass-received.yml.

  This covers all cases:
  - Brand-new players joining for the first time.
  - Existing players who joined before this update (not in compass-received.yml).
  - Players who already own a compass from an older version or staff gift
    (duplicate protection catches them without giving another).

DUPLICATE PROTECTION
  Before giving, iterates player.getInventory().getContents() (all 41 slots:
  storage + armor + off-hand) and calls ItemManager.isWaypointCompass() on
  each item. If any slot contains a valid Pinpoint Compass (PDC-tagged), the
  player is marked received without receiving another compass.

CONFIG CHANGES
  Renamed: newcomer-compass → compass
  Removed key: give-on-first-join (always applies now)
  Added key: inventory-full

  New config block:
    compass:
      enabled: true
      message: "&bYou received a Pinpoint Compass!"
      inventory-full: "&eYour inventory was full. Your Pinpoint Compass was dropped nearby."

PLUGIN WIRING (PinpointPlugin.java)
  compassDataManager initialized and loaded in onEnable() before itemManager.
  compassDataManager.load() called in reload() to pick up any manual edits.
  getCompassDataManager() getter added.

Build Result:
  BUILD SUCCESSFUL — no errors, 1 pre-existing deprecation note (Gradle API)
  Output: build/libs/Pinpoint-1.3.26.jar (108K)

--- FEATURE 1: Newcomer Compass ---

New players (hasPlayedBefore() == false) automatically receive a Pinpoint
Compass 1 tick after their first join. If inventory is full the item drops
naturally at their location. The feature is fully configurable:

  newcomer-compass:
    enabled: true
    give-on-first-join: true
    message: "&aYou received a Pinpoint Compass to help you travel!"

Implementation:
  NewcomerListener.java — listens to PlayerJoinEvent, runs 1 tick later on the
  main thread, calls ItemManager.createWaypointCompass() and adds to inventory.

--- FEATURE 2: Landmarks ---

A new admin-only waypoint type stored alongside regular waypoints in
waypoints.yml with type: LANDMARK. Landmarks are globally accessible (canAccess
always returns true), have no player owner (LANDMARK_OWNER_UUID = UUID(0,0)),
do not appear in player-owned counts, and never trigger the naming flow.

Hologram: a single line parsed with MiniMessage (default gradient style).
GUI: new "Landmarks" tab at slot 50 of the hub bottom nav bar (BEACON icon).
Teleport: uses landmarks.teleport.delay-seconds (default 10) and
           landmarks.teleport.cost (default 0); no owner payment.

Admin Commands (require pinpoint.admin.landmark or sub-permission):
  /wp landmark create <name>   — admin enters placement mode; places block to
                                  set location; auto-receives block if needed
  /wp landmark delete <name>   — removes hologram + storage + block
  /wp landmark list            — shows all landmarks with world/coords
  /wp landmark reload          — reloads config + refreshes landmark holograms
  Alias: /pinpoint now works as an alias for /waypoint

  pinpoint.admin.landmark          (parent, default: op)

SECTION 4 — BEDROCK COMPATIBILITY CONCERNS
================================================================================

The plugin soft-depends on Geyser-Spigot. GuiManager.java contains ZERO
platform-detection code — every GUI is rendered identically for Java and Bedrock.
Geyser translates Java chest inventory events to Bedrock's container UI.

The following concerns apply specifically to Bedrock players:

4.1 LORE VISIBILITY
  Java: lore appears on mouse hover — instantly scannable.
  Bedrock: lore is only visible when an item is actively selected (touched/tapped).
  Impact: status info currently buried in lore (visibility, fee, owner) is
  harder to scan on Bedrock.
  Mitigation in the proposed design: move the most important status into the
  item name where possible. For the Hub Landing, include the count in the
  button name itself ("My Pinpoints (3)") rather than only in lore. For
  category lists, "Free" vs fee amount can appear in lore line 2 — acceptable
  since the player has to interact with the item to teleport anyway.

4.2 CATEGORY SELECTOR (Hub Landing)
  Three large, clearly named buttons on a 3-row chest work very well on Bedrock.
  Bedrock players tap one button → one screen. This is simpler than the current
  flat list where Bedrock players must scroll and read small lore entries.
  The Hub Landing screen actively improves the Bedrock experience.

4.3 GLASS PANE BORDERS
  Render correctly on Bedrock via Geyser. Gray stained glass panes are fine.

4.4 ITEM NAMES AND COLORS
  Adventure API Component text (NamedTextColor.*) passes through Geyser without
  issue. Colors render correctly in Bedrock's container UI.

4.5 CLICK EVENTS
  Geyser maps Bedrock container taps to Java InventoryClickEvent normally.
  getRawSlot() is reliable. No platform-specific click handling is needed.

4.6 PLAYER_HEAD RENDERING
  Bedrock does not render custom player head textures through Geyser in all
  versions. The Invite GUI uses player skulls (makeSkull). On Bedrock, these
  may render as blank/default heads. This is a known Geyser limitation and is
  not new — it exists in the current Invite GUI. The player name label remains
  correct. No action needed now, but noted for future reference.

4.7 DYNAMIC INVENTORY SIZING
  Geyser handles all standard chest sizes (9, 18, 27, 36, 45, 54 slots).
  The dynamic sizing in Invite GUI and Icon Select GUI works on Bedrock.

4.8 ACTION BAR MESSAGES
  player.sendActionBar() works on Bedrock via Geyser. The direction-set
  and fee confirmations that use action bars will appear correctly.

Summary: the proposed category reorganization does not introduce any new
Bedrock compatibility risks. The Hub Landing screen and category lists are
strictly simpler (fewer items on screen, clearer structure) which improves
the Bedrock experience relative to the current dense flat list.

SECTION 5 — CLUTTER AND CLICK REDUCTION SUGGESTIONS
================================================================================

5.1 REMOVE HARDCODED "Category: Mine" FROM HUB LORE  [low effort, immediate]
  GuiManager.java:175 — "Category: Mine" is always shown for owned waypoints
  regardless of icon type. It carries no useful information and contradicts the

5.2 BLOCK RIGHT-CLICK SKIPS HUB ENTIRELY  [medium effort]
  Currently: right-click block → Hub GUI (with slot 49 "This Pinpoint" button)
             → click slot 49 → Manage GUI. That's two clicks.
  Proposed:  right-click block → Manage GUI directly.
  The focusedWaypointId / fromBlock machinery in the Hub exists entirely to

  openUseGui() is the only GUI without fillBorder(). Add one line. Cosmetic
  consistency matters — every other screen has a border.

5.4 REMOVE OR RELOCATE "Get Pinpoint Compass" FROM MANAGE GUI  [low effort]
  Slot 14 gives the player a Pinpoint Compass item. This is a convenience
  shortcut but is unrelated to managing this specific waypoint. It is out of
  place next to Rename, Delete, Change Icon. Options:
    a) Remove it — players can use /wp give or craft it.
    b) Move it to the Hub Landing screen as a small utility button.
  Removing it simplifies the Manage GUI from up to 9 buttons to 8.

5.5 REDUNDANT LORE IN CATEGORY LISTS  [low effort]
  Once waypoints are split into category tabs:
    - "Owner: You" is redundant in My Pinpoints — remove it.
    - "Public" visibility is redundant in the Public tab — remove it.
    - "Private" is useful in the Invited tab — keep it.
  Each lore line removed is one less line Bedrock players must tap through.

  Use GUI is a 3-item screen: display, teleport, back. The teleport is the
  only action a non-owner can take. An alternative: single-click on a
  non-owned waypoint in the category list opens a confirmation/teleport
  immediately (with the waypoint info in a title bar), skipping Use GUI.
  This saves one screen for the most common non-owner action.
  Tradeoff: removes the preview step before a fee-gated teleport. Recommend
  keeping Use GUI for now but deferring this as a future option.

5.7 OFFLINE INVITE REVOCATION  [medium effort, separate feature]
  The current Invite GUI only shows online players. A player who was invited
  while online and then logged off cannot have their access revoked until they
  return online. Proposed fix: add an "Invited Players" section to the Manage
  GUI showing the current invite list (by name, regardless of online status),
  with click-to-revoke per entry. This is separate from the category UX work.

CONFIG FORMAT (config.yml — icon-permissions section)

  Old format (still supported for backwards compatibility):
    icon-permissions:
      pinpoint.icons.media:
        - NOTE_BLOCK
        - JUKEBOX
    → treated as enabled=true, hide-when-locked=false

  New format:
    icon-permissions:
      pinpoint.icons.media:
        enabled: false           # false = group hidden for everyone
        hide-when-locked: true   # true = players without perm see nothing
        icons:
          - NOTE_BLOCK
          - JUKEBOX

  Default config (v1.3.20):
    media:  enabled=false, hide-when-locked=true  → invisible until enabled
    vip:    enabled=false, hide-when-locked=true  → invisible until enabled
    admin:  enabled=true,  hide-when-locked=true  → hidden from non-admins (op has perm)

PARSING LOGIC (GuiManager.java — loadIconGroups)

  Iterates permSec.getKeys(true). For each key:
  - If value is a ConfigurationSection with an "icons" sub-key → new format group.
    Reads enabled, hide-when-locked, icons. Tracks key as a prefix so sub-keys
    (enabled, hide-when-locked, icons) are skipped in subsequent iterations.
  - If value is a List → old format group (backwards compat).
    Key is the permission node; list is the material names.

  Returns List<IconGroup> — one entry per enabled group.

PALETTE BUILDING (buildFullPalette(Player player))

  Starts with base ICON_PALETTE (10 items, always shown, always selectable).
  For each IconGroup:
    - If hide-when-locked=true AND player lacks permission → skip (group not added).
    - Otherwise → add group's icons to palette (some may be locked in buildLockedMap).

LOCKED MAP (buildLockedMap(Player player))

  For each IconGroup where player lacks permission AND hide-when-locked=false:
    marks each material as locked with the group's permission string.
  Locked icons appear in the picker with red "Requires: X" lore and error on click.

COMPATIBILITY
  Old config format (flat list under perm key): still parsed and treated as
  enabled=true, hide-when-locked=false. No config migration required if the old
  format is kept. Behavior change: VIP/Media groups that were previously shown
  locked will be hidden if the server updates config to enabled=false.

  Existing Pinpoint icon selections are never changed — only updated when owner clicks.

STEP 3 — Root cause identified: Bukkit YAML dot-splitting
----------------------------------------------------------
The bug is in GuiManager.buildFullPalette() and buildLockedMap(). Both methods
call permSec.getKeys(false) to iterate the icon-permissions config section.

The config.yml has keys like:
  icon-permissions:
    pinpoint.icons.media:
      - NOTE_BLOCK

Bukkit's YamlConfiguration (via MemorySection.set()) splits ALL dotted path
components when loading YAML. "pinpoint.icons.media" is NOT stored as a flat
key — it is stored as a nested path:
  icon-permissions → pinpoint → icons → media

Confirmed by decompiling MemorySection.class: the set() and get() methods
both use String.indexOf() with '.' as separator and iterate path components.

As a result:
  permSec.getKeys(false)         → ["pinpoint"]  (only the top-level node)
  permSec.getStringList("pinpoint")  → []  (it's a section, not a list)

The inner loop never executes. Zero extra icons are added. buildLockedMap()
has the identical bug — it also uses getKeys(false).

This means the icon-permissions feature has NEVER worked since it was
introduced in v1.3.14. Every reported symptom was consistent with this bug:
correct config, correct JAR, still only 10 base icons.

STEP 4 — Fix
------------
Changed getKeys(false) → getKeys(true) in both buildFullPalette() and
buildLockedMap(). getKeys(true) returns all deep paths including leaf keys:
  ["pinpoint", "pinpoint.icons", "pinpoint.icons.media",
   "pinpoint.icons.vip", "pinpoint.icons.admin"]

For each deep key, getStringList(key) is called:
  - Intermediate section nodes (e.g. "pinpoint", "pinpoint.icons") return []
  - Leaf nodes (e.g. "pinpoint.icons.media") return the actual material list
A `if (matNames.isEmpty()) continue;` guard skips intermediate nodes.

The permission string used in buildLockedMap() is now the full dotted path
(e.g. "pinpoint.icons.media"), which matches:
  - player.hasPermission("pinpoint.icons.media")  ✓
  - iconLockedMessage() switch cases              ✓
  - permToLabel() which splits on "." to get last component  ✓

The fix required no config changes and no YAML format changes.

Project: Pinpoint (Paper 1.21.x plugin, Java 21)

--- WHAT THE PLUGIN DOES ---

Pinpoint is a craftable waypoint and teleportation system.

Players craft a Pinpoint Block (8x Quartz + Ender Eye), place it anywhere,
and name it in chat. The block marks that location as a Pinpoint. Players can
teleport to any Pinpoint they have access to.

Navigation is done via:
1. Pinpoint Compass (1x Ender Eye + 1x Compass) — right-click to open the
   hub GUI listing all accessible Pinpoints.
2. Right-clicking any placed Pinpoint block — opens that block's GUI directly.

Teleports use a countdown (default 10s pearl, 5s block). Moving or taking
damage during the countdown cancels the teleport.

--- SYSTEMS ---

1. Waypoint/Pinpoint system
   - Pinpoint Block: place to mark a location. Named in chat after placing.
   - Owner can: toggle public/private, set fee, invite players, rename,
     delete, change icon, set teleport facing direction.
   - Invited players and public Pinpoints are accessible from the hub GUI.
   - Max waypoints per player configurable (default 10).
   - YAML persistence: plugins/Pinpoint/waypoints.yml (UUID-keyed).

2. Economy / fee system (Vault soft-dependency)
   - Owners can set a fee on their Pinpoint.
   - Balance is checked before the countdown fires.
   - Fee is charged only after player.teleport() succeeds.
   - Fee is paid to the Pinpoint owner via Vault OfflinePlayer API (works
     when the owner is offline).
   - If owner credit fails, the player is refunded and the teleport stands.

3. Teleport flow (TeleportHelper.java)
   - startTeleport / teleportFromBlock: countdown task + cancel listener.
   - doTeleport: world check → location check → balance check → findSafe →
     player.teleport() → charge fee → owner deposit → success messages.
   - Cooldown set after successful teleport.
   - 5-second post-teleport invincibility window.
   - TeleportCancelListener: cancels on player move or damage.

4. Party system
   - Shift+right-click a player with the Pinpoint Compass to send a link
     request. Accepting merges parties (A-B + B-C = A-B-C).
   - Travel notifications: when any party member teleports, all online
     members get a Follow/Stay GUI.
   - /party command (alias /pp): leave, disband, remove, accept, deny,
     follow, stay.
   - Persistence: plugins/Pinpoint/parties.yml.

4a. Teleport facing direction (stored per Pinpoint)
   - Owner sets direction via "Set Facing Direction" button in Manage GUI.
   - Captures both yaw (compass direction) and pitch (vertical angle).
   - Stored as teleport-yaw and teleport-pitch in waypoints.yml.
   - Applied to all teleport paths in TeleportHelper.doTeleport().
   - Pitch description: < -30 = looking up, > 30 = looking down, else level.
   - Float.NaN teleportYaw = direction not set (uses safe-spot default).

4b. Permission-based extra icons (visible-but-locked model) — v1.3.15
   - Base palette (10 items) always visible and usable by all players.
   - Extra icons defined in config.yml under icon-permissions (perm → list).
   - Three built-in groups: pinpoint.icons.media, .vip, .admin.
   - ALL icons from all groups are always shown in the picker regardless of
     the player's permissions.
   - Icons the player cannot use show red lore: "Requires: Media/VIP/Admin".
   - Clicking a locked icon sends an error message; GUI stays open.
     Error messages are configurable in config.yml:
       icon-locked-media, icon-locked-vip, icon-locked-admin, icon-locked-generic.
   - Icon picker inventory is dynamically sized (7 icons/row, 2 border rows).
   - Existing Pinpoint icons are never changed unless the owner clicks.
   - KNOWN ISSUE: Live server config.yml files are missing icon-permissions
     and icon-locked-* keys. See latest entry above for fix.

5. Holograms
   - Invisible marker armor stands above each Pinpoint block showing name,
     owner, visibility, fee.
   - Hidden per-player by distance and line-of-sight.
   - Config: holograms.enabled, holograms.view-distance (default 8),
     holograms.require-line-of-sight (default true), holograms.height
     (default 2.8), holograms.show-owner, holograms.show-fee.
   - KNOWN ISSUE: Live server config.yml files are missing view-distance and
     require-line-of-sight keys. Behavior is correct (uses defaults) but
     config is incomplete.

6. Hub GUI
   - Fixed 6-row, paginated (28 per page).
   - Sorted: owned first (green), invited/private second, public third.
     A-Z within each group.
   - Owned Pinpoints highlighted green with "Owner: You" lore.
   - Each Pinpoint uses its custom icon material.
