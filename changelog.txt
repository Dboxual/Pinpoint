Pinpoint Plugin Changelog
=========================

v1.3.34 — 2026-06-13
Waystone/Landmark/Pinpoint teleport always 5 seconds
-----------------------------------------------------
- Unified both teleport delay paths to 5 seconds.
  Previously: compass/GUI used compass-delay-seconds (default 10s), block used block-delay-seconds (5s).
  Now both compass-delay-seconds and block-delay-seconds default to 5.
- Charge-after-success behavior is unchanged: Bops are only deducted after player.teleport() succeeds.
  If the player moves, takes damage, or cancels during the countdown, no Bops are charged.
- Player-to-player teleport stand-still-seconds is unchanged (separate config key, separate flow).
- Config comment updated to say "Always 5 seconds" for both delay keys.

v1.3.33 — 2026-06-12
Context-aware teleport delay: compass/GUI = 10s, physical block = 5s
---------------------------------------------------------------------
- Teleport delay now depends on HOW the player initiated travel, not just the waypoint type.
- Right-clicking a placed Pinpoint block or Landmark block → 5-second countdown (block-delay-seconds).
- Using the Pinpoint Compass or any GUI tab → 10-second countdown (compass-delay-seconds).
- Both delays apply equally to normal Pinpoints AND Landmarks.
- New top-level config section replaces the old per-path keys:
    teleport:
      compass-delay-seconds: 10
      block-delay-seconds: 5
- Removed: settings.waypoint-pearl-teleport-delay-seconds
- Removed: settings.waypoint-block-teleport-delay-seconds
- Removed: landmarks.teleport.delay-seconds (replaced by the unified teleport section)
- GuiManager now stores a per-player fromBlock flag (set on openHubGui, cleared on full
  GUI close) that is propagated through all sub-GUI navigation so teleport handlers always
  use the correct delay regardless of how deep in the GUI the player is when they click
  "Teleport Here".
- openHubGui previously accepted a fromBlock parameter but silently dropped it. This is
  now stored and used.
- teleportToLandmark() removed from TeleportHelper; landmark teleports route through the
  same teleport() and teleportFromBlock() methods as normal waypoints.
- Player-to-player teleport delay unchanged (stand-still-seconds, default 10s).
- Movement cancel, Bops charging, and cooldown behavior are all unchanged.

v1.3.32 — 2026-06-11
Player-to-player teleport: silent sender, Bops currency, pre-countdown balance check
--------------------------------------------------------------------------------------
- Sending a teleport request is now fully silent for the sender. No chat messages are
  sent to the requester when they click a player in the Players tab.
- Currency renamed from "Bits" / Vault-formatted money to "Bops" across all
  player-to-player teleport GUI lore, chat messages, and config.
- Config key player-teleport.cost-bits renamed to player-teleport.cost-bops.
- Both players must have at least cost-bops Bops BEFORE the countdown starts.
  If either cannot afford it, the accept is rejected and both players are told clearly.
  Previously the check happened post-teleport.
- Charge happens only after the teleport successfully completes (unchanged).
- Post-teleport charge failure (edge case: player spent Bops during countdown) now
  logs a warning to console and safely handles refunds — neither player is double-charged.
- Fixed: renderPlayersContent was reading deleted config key settings.player-teleport-cost.
  Now correctly reads player-teleport.cost-bops.
- Normal waypoint fees (Vault) are unchanged.

v1.3.31 — 2026-06-11
Player-to-player teleport: GUI accept/deny flow
------------------------------------------------
- Accept/deny flow moved from clickable chat to Pinpoint GUI.
  Players open the Pinpoint compass; a pending request banner appears in row 0 of
  the category GUI with Accept (lime) and Deny (red) buttons.
- Request expiry reduced from 30 s to 10 s (player-teleport.request-expire-seconds).
- Added player-teleport config section (request-expire-seconds, stand-still-seconds,
  cost-bops). Old settings.player-teleport-* keys removed.
- Added player-teleport expiry timestamp tracking in WaypointManager for GUI
  "Expires in Xs" display.
- /waypoint tpaccept and /waypoint tpdeny retained as undocumented fallbacks.

v1.3.27 — 2026-06-10
Landmark Block Item (dedicated admin placement block)
------------------------------------------------------
- Added a dedicated Landmark Block item separate from normal Pinpoint Blocks.
  PDC key: pinpoint:landmark_block. Normal Pinpoint Block PDC: pinpoint:waypoint_block.
  The two items are completely distinct — isWaypointBlockItem() never matches a
  Landmark Block and isLandmarkBlock() never matches a normal Pinpoint Block.
- /wp landmark create <name> now gives admins a Landmark Block (not a Pinpoint
  Block) if they don't already have one in their inventory.
- BlockPlaceListener now handles Landmark Blocks and Pinpoint Blocks in separate
  branches. A Landmark Block requires: (1) pinpoint.admin.landmark.create permission,
  (2) active landmark setup mode (set by /wp landmark create). Both conditions must
  be met or placement is cancelled with a configurable message.
- If a Landmark Block is placed without active setup mode, placement is cancelled
  and the setup-required-message is shown.
- If a player without permission places a Landmark Block, placement is cancelled
  and the no-permission-message is shown.
- /wp landmark delete now reads landmarks.block.material from config when checking
  the block type to remove (previously hardcoded to LODESTONE).
- Normal Pinpoint creation logic is completely unchanged.
- Landmark creation never touches the player waypoint limit.
- Config additions under landmarks.block: material, name (MiniMessage), lore
  (legacy & codes), no-permission-message, setup-required-message.

v1.3.26 — 2026-06-10
Automatic Pinpoint Compass Distribution (permanent tracking)
-------------------------------------------------------------
- Replaced first-join hasPlayedBefore() check with a permanent UUID-based
  "received compass" tracking system, stored in compass-received.yml.
- Every player who has never received a Pinpoint Compass gets one automatically
  on their next login: new players, existing players, and anyone who joins after
  this update.
- Duplicate protection: before giving a compass, checks if the player already
  possesses a valid Pinpoint Compass in their inventory. If they do, they are
  marked as received without being given another. Prevents duplication for
  players who already owned a compass from an older version or were given one
  by staff.
- If inventory is full, the compass drops naturally at the player's feet and the
  "inventory-full" message is shown instead of the normal receive message.
- After receiving (or already owning) a compass, the player is permanently
  marked and will never be given one automatically again.
- Added CompassDataManager — mirrors GuideBook's PlayerDataManager pattern.
  Storage: plugins/Pinpoint/compass-received.yml, UUID-keyed, safe for large
  player counts, persistent across restarts.
- Renamed config section newcomer-compass → compass. Removed give-on-first-join
  key (now always applies). Added inventory-full message key.
- Config keys:
    compass.enabled          (boolean, default true)
    compass.message          (default "&bYou received a Pinpoint Compass!")
    compass.inventory-full   (default "&eYour inventory was full...")

v1.3.20 — 2026-06-05
Added — Per-group enabled/hide-when-locked controls for icon-permissions
-------------------------------------------------------------------------
- icon-permissions config groups now support three sub-keys:
    enabled: true/false         — false removes the group entirely; icons
                                  never appear for anyone in the picker.
    hide-when-locked: true/false — true hides icons from players who lack
                                   the permission (no icon shown at all);
                                   false shows icons with red "Requires: X"
                                   lore and an error on click (old behavior).
    icons: [list of materials]  — unchanged from before.
- Default config ships with media and vip disabled (enabled: false) and all
  groups set to hide-when-locked: true. Admin group remains enabled and hidden
  by default (admins already have the permission via op).
- Backwards compatible: old flat-list format (perm key → material list) still
  works and is treated as enabled=true, hide-when-locked=false.
- GuiManager: buildFullPalette() now takes a Player; loadIconGroups() parses
  both old and new config formats; resolveMaterials() helper extracted.
- Players only see icons from groups that are enabled AND (either they have
  the permission OR hide-when-locked is false).

v1.3.18 — 2026-06-05
Changed — Cardinal direction picker integrated into creation flow; legacy yaw auto-migrated
-------------------------------------------------------------------------------------------
- After naming a new Pinpoint, the "Facing Direction" picker GUI opens automatically
  so every Pinpoint gets a cardinal direction set at creation time rather than requiring
  a separate visit to the Manage GUI.
- Old Pinpoints that have a raw teleport-yaw value stored in waypoints.yml are now
  auto-migrated to the nearest cardinal direction on load (no manual action needed):
    yaw   0 / near 0    → SOUTH
    yaw  90              → WEST
    yaw 180              → NORTH
    yaw 270 / near -90   → EAST
  The converted direction is persisted to teleport-direction on the next save; the
  old teleport-yaw / teleport-pitch keys are removed from YAML on save.
- saveWaypoint() simplified: only writes teleport-direction; always nulls old
  teleport-yaw and teleport-pitch keys so they are cleaned from YAML on any save.
- Manage GUI slot 21 ("Facing Direction") no longer has a legacy-yaw display
  branch — all loaded data is cardinal so only hasTeleportDirection() is checked.
- No change to teleport behaviour, cooldowns, fees, invites, party follow,
  hologram, or icon systems.

v1.3.16 — 2026-06-04
Fix — Icon picker extra icons never appeared (Bukkit dot-splitting bug)
-----------------------------------------------------------------------
- buildFullPalette() and buildLockedMap() used getKeys(false), which only
  returned the top-level YAML node "pinpoint" — not the full permission
  paths "pinpoint.icons.media" etc. — because Bukkit's MemorySection splits
  all dotted YAML keys into nested sections on load.
- Fixed by using getKeys(true) to iterate all deep paths, then filtering
  intermediate section nodes with an isEmpty() guard on getStringList().
- No config changes required. The icon-permissions section in config.yml
  was already correct; the bug was entirely in the Java iteration logic.
- The icon-permissions feature has never worked since its introduction in
  v1.3.14; this is the first release where extra icons actually appear.

v1.3.13 — 2026-06-03
Fix — Teleport fee charged after successful teleport, not before
-----------------------------------------------------------------
- Fee is now withdrawn only after player.teleport() succeeds.
- If the teleport is cancelled or interrupted before the countdown ends, no
  fee is charged.
- If no safe landing spot is found and the teleport aborts, no fee is charged
  (removed the now-unnecessary refund block in the safe-spot failure path).

  player is refunded as before — but the teleport is no longer rolled back,
  since the player has already landed.
- Balance check (hasBalance) still runs before the countdown fires so players
  with insufficient funds are turned away immediately.
- All fee amounts, fee calculations, countdown behaviour, restrictions, and
  validation are unchanged.
